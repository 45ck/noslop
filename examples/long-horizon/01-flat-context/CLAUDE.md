# Task Manager API

A REST API for managing tasks, projects, and team assignments. Ships as a Docker container behind an Nginx reverse proxy.

## Tech stack

- Runtime: Node.js 20
- Language: TypeScript 5.4 (strict mode)
- Framework: Express 4
- ORM: Prisma 5 with PostgreSQL 16
- Validation: Zod
- Auth: JWT (access + refresh tokens)
- Testing: Vitest + Supertest
- Linting: ESLint 9 (flat config) + Prettier

## Commands

```bash
npm run dev          # Start dev server with hot reload (port 3000)
npm run test         # Run all tests
npm run test:watch   # Run tests in watch mode
npm run build        # Compile TypeScript to dist/
npm run lint         # ESLint check
npm run lint:fix     # ESLint auto-fix
npm run format       # Prettier format
npm run ci           # lint + test + build (run before every PR)
npm run db:migrate   # Run pending Prisma migrations
npm run db:seed      # Seed dev database
npm run db:reset     # Drop, recreate, migrate, seed
```

## Architecture

```
src/
  routes/          # Express routers — URL mapping only, no logic
  controllers/     # Parse request, call service, send response
  services/        # Business logic — all domain rules live here
  repositories/    # Prisma queries — one file per entity
  middleware/      # Auth, validation, error handling, logging
  utils/           # Pure helper functions
  types/           # Shared TypeScript types and Zod schemas
prisma/
  schema.prisma    # Database schema
  migrations/      # Migration history
  seed.ts          # Dev seed data
```

Request flow: `route → middleware (auth, validate) → controller → service → repository → Prisma → PostgreSQL`

## Code conventions

### Error handling

Throw `AppError` from services. Never throw raw `Error`.

```typescript
throw new AppError('TASK_NOT_FOUND', `Task ${id} not found`, 404);
```

The error middleware catches `AppError` and returns a JSON:API error response. Unhandled errors return 500 with no detail in production.

### Validation

Every route that accepts a body or query params must have a Zod schema in `src/types/`. Apply validation via the `validate` middleware:

```typescript
router.post('/tasks', validate(createTaskSchema), taskController.create);
```

### Auth

Protected routes use the `authenticate` middleware. Role checks use `authorize('admin')`. Public routes (login, register, health) skip both.

```typescript
router.get('/tasks', authenticate, taskController.list);
router.delete('/tasks/:id', authenticate, authorize('admin'), taskController.remove);
```

### Database queries

- Always select specific fields — never `SELECT *` (use Prisma `select`)
- Use transactions for multi-table writes
- Repository methods return plain objects, never Prisma client instances
- Paginate all list endpoints (default 20, max 100)

### Naming

- Files: `kebab-case.ts`
- Classes/types: `PascalCase`
- Functions/variables: `camelCase`
- Database tables: `snake_case` (Prisma maps automatically)
- Routes: plural nouns (`/tasks`, `/projects`, `/users`)

## Testing

### Integration tests (routes)

Use Supertest against a real Express app with a test database. Each test file resets the database in `beforeEach`. Located next to the route file: `src/routes/tasks.test.ts`.

```typescript
it('returns 404 for nonexistent task', async () => {
  const res = await request(app).get('/api/tasks/999').set('Authorization', `Bearer ${token}`);
  expect(res.status).toBe(404);
  expect(res.body.errors[0].code).toBe('TASK_NOT_FOUND');
});
```

### Unit tests (services)

Mock repository functions. Test business logic in isolation. Located next to the service: `src/services/task-service.test.ts`.

### Coverage

Minimum 90% line coverage overall. Services must have 95%+.

## API conventions

- Base path: `/api`
- Versioning: none yet (v1 implied)
- Errors follow JSON:API format: `{ errors: [{ code, title, detail, status }] }`
- List endpoints return: `{ data: T[], meta: { total, page, pageSize } }`
- Single-resource endpoints return: `{ data: T }`
- Dates: ISO 8601 strings
- IDs: UUIDs (generated by the database)
